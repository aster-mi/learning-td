# Skill: 市場調査と改善企画

## 目的

外部の市場情報、競合、ユーザー反応を見て、
改善案や推進案を継続的に企画し、GitHub の task に落とす。

## 先に読むもの

- task state の正本は `.ai/skills/SKILL_GITHUB_TASK_WORKFLOW.md`
- Issue の書き方は `.ai/skills/SKILL_ISSUE_WORKFLOW.md`
- 人判断が必要なら `.ai/skills/SKILL_GITHUB_DISCUSSIONS_WORKFLOW.md`
- プロダクト前提は `.ai/domain/product.md`

## この skill を使うとき

- 市場調査をして改善案を出したい
- 競合比較から企画を起こしたい
- 継続率、導線、報酬、コンテンツ拡張の打ち手を増やしたい
- 集客や再訪の施策案を棚卸ししたい
- 仕様検討の前に、外部の成功例や失敗例を見たい

## 原則

- 市場調査は current な情報が前提なので、必要に応じて最新の公開情報を確認する
- 調査結果は repo 内 Markdown に溜めず、Issue / Discussion に残す
- 1 調査テーマ 1 論点を原則にする
- 調査だけで終わらせず、実行候補は必ず Issue 化する
- タイトル、本文、コメントは原則日本語で書く

## 見る場所

### 1. 外部情報
- 競合プロダクトの store page、公式サイト、公開 changelog
- ユーザーレビュー、公開 FAQ、コミュニティ反応
- 類似ジャンルの onboarding、復帰導線、イベント運用
- マネタイズ、報酬設計、継続導線の公開情報

### 2. 自プロダクト
- open Issue と Done Issue
- 既存 PR と release 履歴
- `.ai/domain/product.md`
- `.ai/domain/architecture.md`
- `.ai/domain/decisions.md`

### 3. 実行結果
- build / test / validate
- 実機確認やプレイ確認で見えた摩擦
- question 品質、進行テンポ、報酬体験の違和感

## 企画テーマの例

- 初回体験の改善
- 継続率改善
- 離脱ポイントの圧縮
- デイリー復帰導線
- 報酬テンポと達成感
- 新規コンテンツ案
- SNS / 配信 / 口コミを意識した推進施策

## フロー

### 1. 調査テーマを決める
- 何を伸ばしたいかを 1 つに絞る
- 例:
  - 初回定着
  - 継続率
  - 問題品質
  - 報酬体験
  - 集客導線

### 2. 外部事例を集める
- 競合や近い体験を 3 件前後見る
- 似ている点と違う点を切り分ける
- 良かった点、避けたい点を分けてメモする
- current 情報が必要な論点では、古い記憶だけで決めない

### 3. 自プロダクトに引き寄せる
- その事例を learning-td にそのまま入れられるかを考える
- 制約を確認する
  - 実装コスト
  - 既存仕様との衝突
  - 世界観やドメインとの相性
  - non-goal との衝突

### 4. 企画候補に分解する
- 調査結果をそのまま貼らず、実行可能な候補に分ける
- 1 候補につき最低限以下を用意する
  - 何を改善 / 推進したいか
  - 根拠になった市場シグナル
  - 期待する変化
  - スコープ
  - 非スコープ
  - リスク

### 5. GitHub に落とす
- 実行可能な候補は Issue にする
- 戦略判断や優先順位判断が必要なら Discussion を立てる
- Project 2 に載せて `Backlog` に置く
- すぐ実装可能な粒度まで固まったら `Ready` に上げる

### 6. 継続運用する
- 調査 1 回で終わらせず、企画候補を継続的に足す
- 実施済み施策の結果を見て、次の調査テーマを更新する
- 有効だった判断だけ `.ai/domain/decisions.md` に残す

## GitHub への残し方

### Issue に残すもの
- 提案の概要
- 背景
- 市場調査から得た示唆
- learning-td 向けの仮説
- 受け入れ条件のたたき台
- 関連リンク

### Discussion に残すもの
- 大きな方向性の選択
- 優先順位の相談
- 実装前に human 判断が必要な論点

## Issue コメント例

```md
市場調査メモ:
- 類似タイトル 3 件を確認
- 共通して初回 3 分以内に達成感を返している
- レビューでは「報酬の見返りが早い」点が好評

企画仮説:
- learning-td でも初回達成報酬を前倒しすると離脱を抑えられる可能性がある

候補:
- 初回ミッション報酬の前倒し
- チュートリアル中の達成演出強化
```

## Discussion 件名例

- `[request][cross-task] 市場調査: 初回定着施策の優先度相談`
- `[decision][issue-28] 市場調査: 報酬テンポ調整の方向性確認`

## やらないこと

- 外部調査メモを repo 内の独自 Markdown に増やす
- 調査結果だけで task 化せず終える
- 競合の施策をそのまま模倣前提で採用する
- current な話を古い記憶だけで決める
