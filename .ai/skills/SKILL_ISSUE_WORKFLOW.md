# Skill: Issue運用

## 目的
GitHub Issue を使って、バグ・改善・問題品質課題を整理し、Codex / Claude / 人間のあいだで追跡しやすくする。

## 基本方針
- 口頭やチャットだけで流さず、再発防止や後追いが必要なものは Issue 化する
- 「今すぐ直す課題」だけでなく、「後で直したい改善案」も Issue 化してよい
- 1 Issue 1テーマを原則にする
- 修正コミットや PR には `Closes #123` などを付けて閉じる

## Issue化すべきもの
- ユーザー影響のある不具合
- UI/UX の違和感や操作しづらさ
- 問題文・選択肢・回答品質の課題
- ステージ難度、報酬、育成コストなどのバランス課題
- 継続率改善や導線改善の提案
- 今回は見送るが忘れたくない改善案

## Issue化しなくてよいもの
- その場で即修正し、追跡価値がほぼない軽微な typo
- 一時的な検証メモだけで十分なもの

## ラベル案
- `bug`
- `feature`
- `question-quality`
- `balance`
- `ux`
- `content`
- `docs`

必要に応じて以下も使う:
- `retention`
- `stage`
- `unit`
- `gacha`

## 優先度
- `P1`: 早めに止血したい。体験を壊す、進行不能、品質事故
- `P2`: 通常優先。近いうちに直したい
- `P3`: アイデア、改善候補、後回し可能

## 記載テンプレ
- 概要
- 背景 / 何が困るか
- 再現手順 or 現状
- 期待する状態
- 優先度
- 関連ファイルや関連Issue

## エージェント運用
- セッション中に新たな課題が見つかったら、必要なら handoff にも「Issue化候補」として残す
- 既知課題に着手する前は、Issue 番号ベースで話すと引き継ぎしやすい
- Skills やポリシー更新が必要な学びは、Issue だけで終わらせず関連ドキュメントにも反映する

## このプロジェクトで特に重要
- 問題品質系は `question-quality`
- 継続率や報酬設計は `retention` / `balance`
- 画面の見切れや導線は `ux`
- ユニット追加や背景追加は `content`
